# Лидерборд — дизайн (2026-07-18)

## Что строим

Вторая вкладка в окне Достижений (кнопка с кубком / клавиша O): «⚡ Лидерборд» —
топ игроков по силе братвы. Читают все, **попадают в топ только игроки из
Telegram Mini App** (подписанный tg-id).

## Решения и обоснования

- **Метрика**: сила братвы, считает СЕРВЕР из валидированной команды снапшота:
  `power = Σ(level + stage×10 + (shiny ? 5 : 0))` по команде (потолок 6×95=570,
  level уже клампится ≤70 валидацией). Дополнительно клиент шлёт `badges`
  (значки арен, кламп 0–99) и `dex` (пойманные виды, кламп 0–999) — колонки
  строки, не влияют на сортировку.
- **Вид строки** (утверждено): компактная таблица — место (🥇🥈🥉/число), ник,
  ⚡сила, 🏅значки, 📖виды. Своя строка подсвечена (матч по `playerNick`).
- **Гейт по tg-id** (утверждено): запись в лидерборд происходит только при
  валидном `initData` (`verifyInitData`, как у покупок). Причина: `POST /team` —
  открытый API без подписи, накрутка curl-ом тривиальна; для лидерборда это
  неприемлемо. Ключ дедупа строки = tg-id (решает и дубли clientId-на-устройство).
  Браузерные игроки без TMA продолжают кормить пул соперников, но в топ не
  попадают (осознанный компромисс).
- **Сейв-свап** (сценарий photoelf ↔ сейв Вани): ник не входит в сейв-код
  (живёт в localStorage/CloudStorage `profile`), поэтому строка лидерборда —
  «последний залитый снапшот под этим tg-id». Поиграл на чужом сейве — после
  первого боя твоя строка покажет чужую братву; вернулся на свой — после
  первого боя вернётся обратно. Самокорректируется, принимаем.
- **Оставшийся вектор накрутки** (принят): валидный initData живёт 24 ч —
  его владелец может слать выдуманные badges/dex и качать снапшот curl-ом.
  Потолки (570 силы, клампы) ограничивают ущерб; для ~33 дружественных
  игроков достаточно.

## Бэкенд (worker.js)

- `POST /team`: принимает опциональные `initData`, `badges`, `dex`. Снапшот и
  пул соперников работают как раньше (без изменений для старых клиентов).
  При валидном initData — upsert в KV-ключ **`lb`** (JSON-массив топ-50
  `{tg, nick, power, badges, dex, ts}`): убрать запись с тем же tg → добавить →
  сортировка по power убыв. → обрезать до 50 → put. Гонки записи не критичны
  (как у `idx`).
- **`GET /leaderboard`** (под `rateLimited`): отдаёт `{top: [{nick, power,
  badges, dex}]}` — tg-id наружу НЕ отдаём. 1 KV-чтение на запрос.
- Сидинга нет: старые снапшоты анонимны (без tg-id), лидерборд наполняется
  по мере боёв TMA-игроков с новым клиентом.

## Клиент

- `net.js`: `netUploadTeam` дополнительно шлёт `initData`, `badges: G.badges.length`,
  `dex: G.dex.caught.size`. Новый `netFetchLeaderboard(cb)` — кэш в памяти 5 мин
  (не жечь KV на переоткрытиях).
- `main.js` + `index.html`: вкладки в `#ach-panel` (`#ach-tabs`, стиль как у
  вкладок лавки), рендер строк классом `.arow`; подсветка своей `.lb-me`.
  Пустой топ → «Сыграй бой в Telegram — и попадёшь в топ!».
- **XSS**: чужие ники рендерим ТОЛЬКО через `textContent` (санитайзер воркера —
  не единственный барьер).
- `sw.js`: поднять версию CACHE.

## Деплой и проверка

Сначала воркер (`env -u CLOUDFLARE_API_TOKEN npx wrangler deploy`), потом фронт
(`git push`). Проверка: `node --check`, локальный `wrangler dev` с dummy
BOT_TOKEN + подписанный питоном initData (upsert lb, гейт без initData,
/leaderboard), прод-smoke после деплоя.

## KV-бюджет

+1 read/+1 write на заливку команды (не чаще раза в 90 с на клиента),
+1 read на открытие вкладки (с клиентским кэшем 5 мин). В лимиты укладываемся.
